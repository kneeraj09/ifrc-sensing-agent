How Signals Are Extracted
  1. Connector fetches raw text from the source
  2. Claude reads each item and extracts structured signals
  3. Each signal records: Location, Commodity, Signal type, Urgency, Confidence (0–1)
  4. Signals stored with full provenance: source, URL, timestamp
  5. Belief states recomputed every cycle over the last 72 hours

Understanding the Demand Index

The demand index is a RELATIVE score (0.0–1.0), not an absolute quantity.

It is a weighted average of signal confidence, adjusted for source veracity and recency of signals.

cautious estimate
P50 Central estimate USE THIS for planning
P90 Upper bound

Source Veracity Weights

Each source is weighted by institutional credibility. IFRC GO (0.95) carries the highest trust as IFRC's own first-party operational data. Signals from GDACS or ReliefWeb carry nearly twice the weight of a GDELT media article. The demand index reflects this automatically — one IFRC GO field report outweighs a dozen media headlines.

Key Rules for Interpretation
  • Index is RELATIVE — compare locations to each other, not to a fixed threshold
  • Check the signal count — P50 of 0.8 from 1 signal is weaker than 0.6 from 8
  • Signals decay over 48 hours — recent signals always outweigh older ones
  • Human overrides take precedence — field officers can annotate belief states
  • SCQA briefing refreshes every 30 min — reflects the last full ingestion cycle
Baseline Layer Hazard Baseline — Historical Risk & Response 5-year GDACS disaster archive linked to IFRC GO operations via GLIDE numbers
What the Hazard Baseline Is

The Hazard Baseline is a static analytical layer — separate from the real-time signal pipeline — that provides historical context for each country. It answers: "How often does this country get hit, by what type of disaster, and did IFRC respond?"

It combines two data sets:

  • GDACS historical — 5 years of earthquake, flood, cyclone, drought, wildfire, and volcano events via the GDACS REST API, filtered to IFRC priority regions
  • IFRC GO operations — 2 years of emergency events and appeals from the IFRC GO platform, cross-referenced via GLIDE numbers
How to Build the Baseline (Two Steps)
  1. Go to the Hazard Baseline dashboard (/hazard)
  2. Click ⬇ Load GDACS History — fetches 5 years of Africa disaster events (~30s). Progress shows in the log panel below the button
  3. Once done, click 🔗 Link IFRC GO — fetches 2 years of IFRC GO emergency operations and matches them to GDACS events (~20s)
  4. Page reloads automatically — the country table populates with severity profiles and IFRC GO response coverage
  5. Run again periodically (monthly recommended) to capture new events

GLIDE Number — The Linking Key
GLIDE (Global IDEntifier) is a shared disaster reference code adopted across humanitarian systems. When a cyclone hits Mozambique, GDACS, IFRC GO, UN OCHA, and national civil protection all reference the same GLIDE: e.g. TC-2023-000056-MOZ. This is the primary key that links GDACS historical events to IFRC GO operations.
  • GLIDE coverage is highest for large-scale operations (Emergency Appeals, DREF activations)
  • Smaller events may lack a GLIDE on one side — the system falls back to iso3 country matching
  • The N GLIDE tag in the hazard table shows exact event-level matches (not just country)

Gap Analysis

The gap analysis table appears automatically below the country profiles whenever red-alert GDACS events exist for a country but no matching IFRC GO operation is found. This is a signal for potential under-response — not a definitive finding.

  • Gaps can be genuine (no IFRC response) or artefacts (GO operation exists but lacks GLIDE metadata)
  • The 2-year GO window vs 5-year GDACS window means older events will always show as gaps
  • Use the gap table as a prompt for investigation, not a conclusion
  • Cross-check against National Society reports and regional operation records before acting

Agent 2 Demand Aggregation & Network Shaping Turns incoming logistics requests into consolidated shipment proposals
How Requests Enter the System
  1. Field staff send a logistics request by email, WhatsApp, or manual form in the Demand dashboard
  2. Claude reads each message and extracts a structured request (origin, destination, commodity, quantity, deadline)
  3. Requests with confidence < 0.5 are discarded; the rest are stored as pending
  4. Run the demand cycle to cluster pending requests and generate proposals
  5. Coordinator reviews each proposal: Accept, Decline, or Modify with notes

How Clustering Works

The agent groups pending requests by commodity class (food, medical, NFIs…) and destination. A cluster forms when 2 or more compatible requests share the same corridor. Claude then generates a plain-language consolidation proposal for each cluster.

  • Commodity class is normalised — "rice" and "food rations" both map to food
  • Destination matching is case-insensitive ("Juba" = "juba")
  • Requests with different units (MT vs. cartons) can still be clustered if compatible
  • Clusters are rebuilt fresh each time the demand cycle runs — stale clusters are cleared first
Consolidation Proposals

Each proposal contains:

ElementWhat it tells you
Proposal textPlain-language summary of the consolidation opportunity
RationaleWhy these requests can be combined (same corridor, commodity, timing)
Estimated savingApproximate cost or effort reduction from combining shipments

Reviewing a Proposal
  1. Accept — marks the cluster as accepted; constituent requests move to accepted status
  2. Decline — dismisses the proposal; requests remain pending for future cycles
  3. Modify — accept with coordinator notes that override or qualify the AI recommendation

Manual requests always have confidence 1.0 and override extracted requests in ranking. Add coordinator notes to any proposal to create an audit trail for the decision.

Agent 3 Allocation & Ethical Tradeoff Newsvendor algorithm · Regional stock pools · 5 ethical scenarios · Human ratification
Regional Stock Pools

Stock is regionally segmented. Africa stock only allocates to Africa locations.

When adding a stock position, assign it to a region. The newsvendor model runs independently per region, then results are merged. A location with no matching regional stock pool will not appear in the allocation output.

RegionHow to assign
Africa Asia Middle East
Latin America Europe North America
Oceania Global 		Select from the Region dropdown when entering a stock position. Use Global for unallocated reserve stock.

Region is auto-detected for belief states from the country name. Stock regions must be set manually by the coordinator.

The Newsvendor Algorithm

The agent uses a multi-location newsvendor model to compute how to allocate a fixed regional stock across multiple locations under demand uncertainty. It uses the P10/P50/P90 demand estimates from Agent 1 as inputs.

Optimal quantity per location: Q* = μ + σ × Φ⁻¹(critical ratio)

  • μ = P50 demand estimate (central case)
  • σ = demand spread, derived from P10–P90 range
  • Critical ratio = the ethical stance on risk — higher means fewer shortfalls
  • An equity floor guarantees every location receives at least its P10 estimate
  • If total Q* exceeds regional stock, residuals are scaled down proportionally after floors are paid

Coordinator Workflow
  1. Go to the Allocation dashboard and add your stock position(s) — set the Region for each depot entry
  2. Enter the commodity and unit you want to allocate, then click Run allocation
  3. The system groups stock and belief states by region and runs the newsvendor model independently per region
  4. All 5 scenarios are computed simultaneously — compare the scenario cards: equity score, average coverage, locations at risk
  5. Click a scenario card to select it — Claude generates a 3-paragraph decision brief
  6. Optionally enter manual overrides for specific locations (your field judgement takes precedence)
  7. Review the brief: Situation, Ethical Rationale, Residual Risk
  8. Ratify the decision with a brief justification — this locks the run and creates an audit record
The Decision Brief

Claude generates a 3-paragraph brief for the selected scenario:

ParagraphContent
SituationWhat is being allocated, to whom, and under which scenario
Ethical RationaleTradeoffs made explicit — who benefits most, who is at risk of shortfall, alignment with the impartiality principle
Residual RiskWhat could go wrong, assumptions being made, what to monitor — flags any location with > 30% shortfall probability

The brief ends with one sentence the coordinator can use to justify the decision to stakeholders. Coordinator overrides are flagged explicitly in the brief.

Coordinator Overrides
  • Enter a specific quantity for any location in the override fields after selecting a scenario
  • Overrides are visible in the decision brief and audit trail — they are not hidden
  • If an override reduces a location's allocation below its equity floor, the brief will flag this
  • Use overrides when you have field intelligence the system cannot access (e.g. access route closed, stock already pre-positioned)
Key Principles
  • The algorithm is a decision support tool, not a decision-maker — the coordinator always ratifies
  • Stock is regionally pooled — enter a Region for every depot to ensure correct scoping
  • Allocation uses demand estimates from Agent 1 — run ingestion first for current data
  • A ratified run is locked; a new run must be started to revise an allocation
  • All runs are stored — the run history is accessible at the bottom of the Allocation dashboard
  • No allocation is distributed until explicitly ratified by a coordinator
