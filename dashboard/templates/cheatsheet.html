How Signals Are Extracted
  1. Connector fetches raw text from the source
  2. Claude reads each item and extracts structured signals
  3. Each signal records: Location, Commodity, Signal type, Urgency, Confidence (0–1)
  4. Signals stored with full provenance: source, URL, timestamp
  5. Belief states recomputed every cycle over the last 72 hours

Understanding the Demand Index

The demand index is a RELATIVE score (0.0–1.0), not an absolute quantity.

It is a weighted average of signal confidence, adjusted for source veracity and recency of signals.

Source Veracity Weights

Each source is weighted by institutional credibility. IFRC GO (0.95) carries the highest trust as IFRC's own first-party operational data. Signals from GDACS or ReliefWeb carry nearly twice the weight of a GDELT media article. The demand index reflects this automatically — one IFRC GO field report outweighs a dozen media headlines.

Key Rules for Interpretation
  • Index is RELATIVE — compare locations to each other, not to a fixed threshold
  • Check the signal count — P50 of 0.8 from 1 signal is weaker than 0.6 from 8
  • Signals decay over 48 hours — recent signals always outweigh older ones
  • Human overrides take precedence — field officers can annotate belief states
  • SCQA briefing refreshes every 30 min — reflects the last full ingestion cycle
Agent 2 Demand Aggregation & Network Shaping Turns incoming logistics requests into consolidated shipment proposals
How Requests Enter the System
  1. Field staff send a logistics request by email, WhatsApp, or manual form in the Demand dashboard
  2. Claude reads each message and extracts a structured request (origin, destination, commodity, quantity, deadline)
  3. Requests with confidence < 0.5 are discarded; the rest are stored as pending
  4. Run the demand cycle to cluster pending requests and generate proposals
  5. Coordinator reviews each proposal: Accept, Decline, or Modify with notes

How Clustering Works

The agent groups pending requests by commodity class (food, medical, NFIs…) and destination. A cluster forms when 2 or more compatible requests share the same corridor. Claude then generates a plain-language consolidation proposal for each cluster.

  • Commodity class is normalised — "rice" and "food rations" both map to food
  • Destination matching is case-insensitive ("Juba" = "juba")
  • Requests with different units (MT vs. cartons) can still be clustered if compatible
  • Clusters are rebuilt fresh each time the demand cycle runs — stale clusters are cleared first
Consolidation Proposals

Each proposal contains:

Reviewing a Proposal
  1. Accept — marks the cluster as accepted; constituent requests move to accepted status
  2. Decline — dismisses the proposal; requests remain pending for future cycles
  3. Modify — accept with coordinator notes that override or qualify the AI recommendation

Manual requests always have confidence 1.0 and override extracted requests in ranking. Add coordinator notes to any proposal to create an audit trail for the decision.

Agent 3 Allocation & Ethical Tradeoff Newsvendor algorithm · 5 ethical scenarios · Human ratification
The Newsvendor Algorithm

The agent uses a multi-location newsvendor model to compute how to allocate a fixed stock across multiple locations under demand uncertainty. It uses the P10/P50/P90 demand estimates from Agent 1 as inputs.

Optimal quantity per location: Q* = μ + σ × Φ⁻¹(critical ratio)

  • μ = P50 demand estimate (central case)
  • σ = demand spread, derived from P10–P90 range
  • Critical ratio = the ethical stance on risk — higher means fewer shortfalls
  • An equity floor guarantees every location receives at least its P10 estimate
  • If total Q* exceeds available stock, residuals are scaled down proportionally after floors are paid

Coordinator Workflow
  1. Go to the Allocation dashboard and enter available stock, commodity, and unit
  2. Click Run allocation — all 5 scenarios are computed simultaneously from Agent 1 belief states
  3. Compare the 5 scenario cards: equity score, average coverage, locations at risk
  4. Click a scenario card to select it — Claude generates a 3-paragraph decision brief
  5. Optionally enter manual overrides for specific locations (your field judgement takes precedence)
  6. Review the brief: Situation, Ethical Rationale, Residual Risk
  7. Ratify the decision with a brief justification — this locks the run and creates an audit record
The Decision Brief

Claude generates a 3-paragraph brief for the selected scenario:

ParagraphContent
SituationWhat is being allocated, to whom, and under which scenario
Ethical RationaleTradeoffs made explicit — who benefits most, who is at risk of shortfall, alignment with the impartiality principle
Residual RiskWhat could go wrong, assumptions being made, what to monitor — flags any location with > 30% shortfall probability

The brief ends with one sentence the coordinator can use to justify the decision to stakeholders. Coordinator overrides are flagged explicitly in the brief.

Coordinator Overrides
  • Enter a specific quantity for any location in the override fields after selecting a scenario
  • Overrides are visible in the decision brief and audit trail — they are not hidden
  • If an override reduces a location's allocation below its equity floor, the brief will flag this
  • Use overrides when you have field intelligence the system cannot access (e.g. access route closed, stock already pre-positioned)
Key Principles
  • The algorithm is a decision support tool, not a decision-maker — the coordinator always ratifies
  • Allocation uses demand estimates from Agent 1 — run ingestion first for current data
  • A ratified run is locked; a new run must be started to revise an allocation
  • All runs are stored — the run history is accessible at the bottom of the Allocation dashboard
  • No allocation is distributed until explicitly ratified by a coordinator
